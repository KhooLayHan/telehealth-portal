# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Stack Overview

Full-stack telehealth platform:
- **Frontend:** React 19 + TypeScript + Vite + TanStack Router/Query/Form + shadcn/ui + Tailwind CSS 4
- **Backend:** ASP.NET Core 10 Minimal APIs, Vertical Slice Architecture, EF Core + PostgreSQL 18, NodaTime
- **Testing:** Vitest + Playwright (frontend); TUnit + TestContainers (backend)
- **Linting:** Biome via Ultracite (frontend/JSON); CSharpier (C#)
- **Package manager:** Bun (frontend + root scripts); .NET CLI (backend)
- **Local infra:** Docker Compose (PostgreSQL, LocalStack for AWS emulation, Seq for logs at port 8081)
- **Lambda functions:** appointment-reminder, lab-pdf-processor (under `/functions/`)
- **IaC:** Pulumi C# (under `/infra/`)

## Development Commands

### Root (runs both layers)
```bash
bun dev                  # Start frontend + backend concurrently
bun build                # Build both
bun test                 # Frontend unit tests + backend tests
bun x ultracite fix      # Auto-fix all frontend/JSON formatting
bun x ultracite check    # Check frontend/JSON formatting
```

### Frontend (`cd frontend`)
```bash
bun run dev              # Vite dev server (http://localhost:5173)
bun run build            # tsc -b && vite build
bun run check            # Biome lint/format check
bun run fix              # Auto-fix with Biome
bun run test             # Vitest unit tests
bun run test:browser     # Playwright browser tests
bun run coverage         # Coverage report
bun run generate:api     # Regenerate API client from OpenAPI spec (Orval)
bun run storybook        # Storybook dev server (port 6006)

# Run a single test file or by name
bun run test -- src/components/Button.test.tsx
bun run test -- --test-name-pattern="should render"
```

### Backend (`cd backend`)
```bash
bun run dev          # dotnet run --project src/TeleHealth.Api (API at http://localhost:5000, Scalar docs at /scalar/v1)
bun run build        # dotnet build
bun run restore      # dotnet restore --locked-mode
bun run check        # dotnet csharpier check .
bun run format       # dotnet csharpier format .
bun run test         # All unit tests

# Run specific tests
dotnet test tests/TeleHealth.UnitTests
dotnet test --filter "FullyQualifiedName~BasicTests"
dotnet test --filter "Name~Add_ReturnsSum"
```

> **IMPORTANT:** Never run `dotnet run` directly from `backend/` вЂ” it will fail with "Couldn't find a project".
> Always use `bun run dev` from `backend/`, which passes `--project src/TeleHealth.Api/TeleHealth.Api.csproj` automatically.

### Local Setup
```bash
cp .env.example .env
docker compose -f docker/docker-compose.yml up -d   # PostgreSQL, LocalStack, Seq
cd backend && bun run dev    # DB migrations run automatically on startup
cd frontend && bun install && bun run dev
```

## Architecture

### Backend вЂ” Vertical Slice

Each feature is a self-contained slice: endpoint handler, request/response DTOs, FluentValidation validator, and domain logic all live together in `Features/<Domain>/<Action>/`. New features should follow this same pattern.

`TeleHealth.Contracts` holds MassTransit event types shared across the API and Lambda functions (e.g., `AppointmentBookedEvent`, `PatientRegisteredEvent`).

### Frontend вЂ” File-Based Routing

TanStack Router generates `routeTree.gen.ts` automatically вЂ” never edit it manually. It regenerates on `bun run dev` / `bun run build`.

Route file naming conventions:
- `_auth.tsx` / `_protected.tsx` вЂ” layout routes (pathless, underscore prefix)
- Files inside `_auth/` require unauthenticated state; files inside `_protected/` require authentication

Key patterns:
- **Server state:** TanStack Query (React Query)
- **Client state:** Zustand (`store/useAuthStore.ts`)
- **Forms:** TanStack Form
- **API client:** Auto-generated by Orval from backend OpenAPI spec вЂ” run `bun run generate:api` after backend changes
- **No barrel files** вЂ” avoid `index.ts` re-export files

## Mandatory Formatting Workflow

These steps are **required** вЂ” Biome and CSharpier treat formatting violations as errors that will block CI.

### Frontend (run from `frontend/`)

| When | Commands |
|------|----------|
| **Before starting work** on any frontend file | `bun run format` в†’ `bun run check` |
| **After finishing** any frontend file | `bun run format` в†’ `bun run check` |
| **Before every commit** | `bun run format` в†’ `bun run check` |

```bash
cd frontend
bun run format   # Biome auto-formats all files (write mode)
bun run check    # Ultracite lint + format check вЂ” must pass with zero errors
```

Both commands must pass clean before you commit. If `bun run check` reports errors, fix them and re-run.

### Backend (run from `backend/`)

| When | Commands |
|------|----------|
| **After creating or editing** any C# file | `bun run format` в†’ `bun run check` |
| **Before every commit** | `bun run format` в†’ `bun run check` |

```bash
cd backend
bun run format   # CSharpier auto-formats all C# files
bun run check    # CSharpier check вЂ” must pass with zero errors
```

Use `/format` (custom Claude skill) to run the correct sequence for whichever layers you touched.

## Code Style

### UI & Design
- **Always use shadcn/ui components** for all UI elements (buttons, inputs, dialogs, cards, badges, tables, etc.) вЂ” import from `@/components/ui/`. Never build custom UI primitives when a shadcn/ui component exists.
- **Always use Framer Motion** for animations and transitions (page transitions, enter/exit, hover effects, loading states) вЂ” import from `framer-motion`. Do not use CSS keyframes or other animation libraries for anything Framer Motion can handle.

### TypeScript / React
- Biome (via Ultracite) enforces formatting; run `bun run format && bun run check` (from `frontend/`) before committing
- Arrow functions for callbacks; `for...of` over `.forEach()`
- `async/await` over promise chains; always `await` in async functions
- `const` by default, `let` only when needed, never `var`
- Optional chaining (`?.`), nullish coalescing (`??`)
- React 19: refs as props вЂ” do not use `forwardRef`
- No components defined inside other components
- No `console.log`, `debugger`, or `alert` in committed code
- No `.only` or `.skip` in tests

### C# / .NET
- CSharpier enforces formatting; run `bun run format && bun run check` (from `backend/`) after every file and before committing
- File-scoped namespaces; primary constructors where appropriate
- `_camelCase` for private instance fields; `s_camelCase` for private static fields
- Warnings treated as errors; nullable reference types and implicit usings enabled
- EF Core uses snake_case naming convention (auto-applied via Npgsql)

## Testing Patterns

### Backend (TUnit)
```csharp
[Test]
public async Task TestName()
{
    await Assert.That(result).IsEqualTo(expected);
}
```
Use `[Before(Class)]`/`[After(Class)]` for class-level setup; `[Before(Test)]`/`[After(Test)]` for per-test hooks. Use AwesomeAssertions for assertions, NSubstitute for mocking.

### Frontend (Vitest)
Use `it()`/`test()` with `async/await`; keep `describe` nesting shallow. No `.only` or `.skip` in commits.

## PII / PHI Rules (Healthcare Compliance вЂ” HIPAA/GDPR)

**Never log, expose in API responses, or include in exception details:**
- Email addresses
- IC numbers (PHI вЂ” never log even for debugging duplicate-registration events)
- Full name + any other identifier
- Phone numbers
- Date of birth + any other identifier
- Passwords or password hashes

**Safe to log:** `PublicId` (GUIDs), Slug values, appointment/record IDs

```csharp
// WRONG
Log.Warning("Login failed for: {Email}", cmd.Email);
throw new ConflictException($"Email '{email}' is already registered.");

// CORRECT
Log.Warning("Login failed вЂ” account not found.");
throw new DuplicateEmailException(); // generic detail message only
```

Auth failures ("user not found" vs "wrong password") must produce the same generic log message вЂ” distinguishing them is an enumeration risk.

API exception `detail` fields surface in ProblemDetails responses (RFC 7807), so they follow the same rules as logs.
